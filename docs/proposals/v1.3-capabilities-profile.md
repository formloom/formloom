# v1.3 — Capabilities Profile

| | |
|---|---|
| **Status** | Proposed |
| **Target release** | v1.3 (minor) |
| **Affects** | `@formloom/schema`, `@formloom/llm` |
| **Breaking changes** | None |
| **Schema format** | Unchanged (`1.2`) |
| **Discussion** | Comments on this PR |

## Summary

Let a host declare **which field types and features the LLM may emit on a given surface**, drive the system prompt, tool JSON Schema, and validator from that single declaration, and reject schemas that violate it. Zero-config behavior stays byte-identical to v1.2; this is purely additive.

## Motivation

Today, the system prompt teaches every primitive and every feature; the tool schema allows every property; the validator accepts every shape the grammar admits. The LLM is free to emit anything. Hosts that want to constrain what's emitted on a given surface have three options today, all bad:

- **Filter post-parse.** Reject schemas that use unwanted features. Problem: the LLM already spent tokens generating the broken schema, and the user sees "something went wrong" instead of a form.
- **Ignore-and-render-partial.** Drop unsupported field types at render time. Problem: silently missing fields confuse users; forms feel broken.
- **Fork the system prompt.** Maintain a modified `FORMLOOM_SYSTEM_PROMPT` in the host codebase. Problem: drifts every time Formloom ships a minor.

The real need is **telling the model up-front what your renderer supports**, and trusting that guidance (with a validator fallback for the 1% of cases where the LLM ignores it).

### Concrete cases

- **Opsflow's four surfaces** (builder intake, agent execution intake, mid-run HITL, public shared pages) each have different UX contracts. The builder might want the full surface; public pages want no `file`, no `readOnly` (it's an intake, not a review), and only `variant: "tool-select"` for their custom widget.
- **Multi-tenant SaaS**. Customer A supports file uploads; Customer B doesn't. Today the host has to branch inside the renderer or post-filter. Capabilities let the branch live at the prompt/tool layer.
- **Regulated contexts**. A healthcare host can set `features.file: false` so the LLM never generates a file upload field; this is a compliance win (no PHI-bearing base64 slipping through).
- **Token budget**. The v1.2 system prompt is ~3.5 KB. A host that only uses text + select saves ~40% by stripping the boolean/radio/date/number/file bullets and the showIf/sections/allowCustom/variant sections.

### Non-motivations

- Not "better validation." The validator already rejects malformed schemas. Capabilities constrain *well-formed* schemas to a host-specific subset.
- Not replacing `showIf`. Hosts that want `showIf` keep using it. Capabilities only let hosts say *"my renderer doesn't do showIf, so the LLM shouldn't emit it."*

## Design

### One declaration, three gates

```
                FormloomCapabilities
                        │
      ┌─────────────────┼─────────────────┐
      ▼                 ▼                 ▼
  Prompt gate       Tool gate        Validator gate
  (soft: LLM       (hard: JSON    (backstop: reject
  guidance)         Schema enum)     or drop)
```

- **Prompt gate** — the system prompt the LLM sees only mentions allowed features. Saves tokens, reduces distraction, improves generation quality for constrained surfaces.
- **Tool gate** — `FORMLOOM_PARAMETERS` is narrowed: `fields[].type.enum` is restricted, disabled-feature properties are deleted, variant allowlists become `enum` on `hints.variant`. Providers with strict structured-output modes (OpenAI response_format, Gemini controlled generation) refuse to emit anything outside the schema.
- **Validator gate** — `validateSchema(input, { capabilities })` catches anything that slipped through text-mode, copy-paste, or an LLM that ignored the tool schema. Same `strict` / `lenient` semantics as the existing `forwardCompat` option.

### The type

```ts
// @formloom/schema
export interface FormloomCapabilities {
  /** Allowed field types. Omit to allow all seven primitives. */
  fieldTypes?: readonly FieldType[];

  /** Feature toggles. Omit a key = allow; set to false = forbid. */
  features?: {
    showIf?: boolean;
    sections?: boolean;
    allowCustom?: boolean;
    optionDescriptions?: boolean;
    readOnly?: boolean;
    disabled?: boolean;
  };

  /**
   * Variant policy.
   *   omit     = any string (v1.2 default)
   *   string[] = allowlist; other variants rejected
   *   false    = no hints.variant allowed at all
   */
  variants?: readonly string[] | false;

  /** Hard cap on total fields per schema. Omit for no cap. */
  maxFields?: number;

  /** Hard cap on options per radio/select. Omit for no cap. */
  maxOptions?: number;
}

export const FULL_CAPABILITIES: FormloomCapabilities;
```

**Omit = allow** throughout. This is critical: an empty `FormloomCapabilities` object is equivalent to not passing capabilities at all. The default path of least resistance is permissive. Hosts opt into constraints one key at a time.

### The factory

```ts
// @formloom/llm
export function createFormloomCapabilities(
  caps: FormloomCapabilities,
): FormloomCapabilitiesBundle;

interface FormloomCapabilitiesBundle {
  capabilities: FormloomCapabilities;
  systemPrompt: string;
  textPrompt: string;
  parameters: unknown;                // narrowed FORMLOOM_PARAMETERS
  tool: { openai, anthropic, gemini, mistral, ollama };
  responseFormat: unknown;            // OpenAI response_format wrapper
  parse(input: unknown, opts?: { forwardCompat?: "strict" | "lenient" }): ParseResult;
}
```

One function, one bundle, every downstream artifact already specialized. The factory is pure (no memoization), no I/O; consumers cache if they need to.

### Host usage — one surface

```ts
import { createFormloomCapabilities } from "@formloom/llm";

const intake = createFormloomCapabilities({
  fieldTypes: ["text", "select", "boolean"],
  features: { showIf: false, allowCustom: true },
  variants: ["combobox"],
  maxFields: 7,
});

const response = await openai.chat.completions.create({
  messages: [{ role: "system", content: intake.systemPrompt }, ...],
  tools: [intake.tool.openai],
});

const toolCall = response.choices[0].message.tool_calls?.[0];
const result = intake.parse(toolCall.function.arguments);
if (result.success) renderForm(result.schema);
```

### Host usage — multiple surfaces

```ts
const intake = createFormloomCapabilities({ fieldTypes: ["text", "select"] });
const clarification = createFormloomCapabilities({
  fieldTypes: ["text", "boolean"],
  features: { sections: false },
});
const publicPage = createFormloomCapabilities({
  features: { file: false /* ...readOnly stays on for recap render */ },
});
```

Runtime profile switching is just picking which bundle to hand to the model.

## Enforcement table

| Capability | Strict error path | Lenient action |
|---|---|---|
| `fieldTypes` doesn't include `field.type` | `fields[N].type` | drop field |
| `features.showIf === false` and field has `showIf` | `fields[N].showIf` | strip `showIf`, keep field |
| `features.sections === false` and schema has `sections` | `sections` | strip `sections` block |
| `features.allowCustom === false` and field has `allowCustom: true` | `fields[N].allowCustom` | set to `false` |
| `features.optionDescriptions === false` and any option has `description` | `fields[N].options[M].description` | strip descriptions |
| `features.readOnly === false` and field has `readOnly: true` | `fields[N].readOnly` | strip flag |
| `features.disabled === false` and field has `disabled: true` | `fields[N].disabled` | strip flag |
| `variants === false` and field has `hints.variant` | `fields[N].hints.variant` | strip variant |
| `variants` is array and field's variant not in list | `fields[N].hints.variant` | strip variant |
| `maxFields` exceeded | `fields` | truncate from end |
| `maxOptions` exceeded on a field | `fields[N].options` | truncate from end |

Lenient mode builds a mutated copy for reporting; the caller's original is never touched. Same discipline as today's unknown-type handler.

## Backward compatibility

| Surface | Before v1.3 | With v1.3, no caps passed | With v1.3, caps passed |
|---|---|---|---|
| `FORMLOOM_SYSTEM_PROMPT` | current value | byte-identical | narrowed |
| `FORMLOOM_PARAMETERS` | current value | deep-equal | narrowed |
| `FORMLOOM_TOOL_OPENAI` etc. | current value | deep-equal | narrowed |
| `parseFormloomResponse` | current behavior | identical | via `bundle.parse` |
| `validateSchema(input)` | current behavior | identical | validates against caps |

A regression test asserts `buildSystemPrompt(FULL_CAPABILITIES) === FORMLOOM_SYSTEM_PROMPT` at the string-equality level, and `narrowParameters(FULL_CAPABILITIES)` deep-equals the current `FORMLOOM_PARAMETERS`. If either drifts, the PR fails.

## Alternatives considered

### Single boolean flags on `validateSchema`

```ts
validateSchema(input, { allowFile: false, allowShowIf: false, ... });
```

Rejected. Doesn't generalize to the prompt or tool schema — the host still has to maintain a parallel system prompt. Capabilities as a first-class type lets one declaration drive all three gates.

### A plugin system for rendering

Rejected as over-engineering. Formloom's contract with the host is "renderer decides everything visual." Adding a plugin registry inverts that. Capabilities are a *declaration of host support*, not an extension mechanism.

### Per-field overrides

```ts
{ fieldTypes: { text: true, file: false } }
```

Rejected. Encourages fine-grained sub-restrictions ("text must have pattern") that explode the matrix. The current allowlist-array form keeps the API flat.

### Runtime capability handshake

Client tells server "I support X, Y, Z"; server fetches the right capabilities. Out of scope — this is a wire-protocol concern, not a library concern. Hosts can build this on top of `createFormloomCapabilities`.

### Generated per-capability types

`createFormloomCapabilities({ fieldTypes: ["text"] })` could return a bundle whose `parse()` result has `schema.fields[N].type` typed as `"text"`. Tempting but complex (requires conditional generic types) for marginal value. Deferred.

## Out of scope (deferred to future minors)

| Deferred | Ship when… |
|---|---|
| Per-field-type sub-restrictions (e.g. "text must have pattern") | A concrete use case shows up |
| Profile composition (`extend` / `merge`) | A second host asks |
| Runtime profile switching mid-conversation | Same-conversation capability shifts become real |
| React-side capabilities (hook rejects schemas that violate host caps) | Renderer-level enforcement becomes necessary |
| Zod adapter capability filtering | Capabilities leak into server validation in practice |
| Client → server capability handshake | Protocol extension project, not a library feature |
| Capability inheritance from a global default | Users request it over explicit bundles |

## Implementation sketch

**New files**

- `packages/schema/src/capabilities.ts` — type + `FULL_CAPABILITIES` + internal check helpers.
- `packages/llm/src/prompt-fragments.ts` — named prompt fragments + `buildSystemPrompt` / `buildTextPrompt`.
- `packages/llm/src/capabilities.ts` — `createFormloomCapabilities` factory + bundle type.
- Tests: `packages/schema/src/__tests__/capabilities.test.ts`, `packages/llm/src/__tests__/capabilities.test.ts`, `packages/integration-tests/src/__tests__/capabilities-e2e.test.ts`.

**Modified files**

- `packages/schema/src/validate.ts` — add `capabilities` to `ValidateOptions`; insert checks per the enforcement table.
- `packages/schema/src/index.ts` — export `FormloomCapabilities`, `FULL_CAPABILITIES`.
- `packages/llm/src/prompt.ts` — `FORMLOOM_SYSTEM_PROMPT` becomes `buildSystemPrompt(FULL_CAPABILITIES)`.
- `packages/llm/src/parameters.ts` — add `narrowParameters(caps)`; `FORMLOOM_PARAMETERS` constant is unchanged.
- `packages/llm/src/index.ts` — export factory, bundle type, and power-user helpers.
- `examples/provider-free/` — capability preset selector for the demo.
- `packages/schema/README.md`, `packages/llm/README.md`, root `README.md` — docs.

## Testing

- **Regression gates.** `buildSystemPrompt(FULL_CAPABILITIES) === FORMLOOM_SYSTEM_PROMPT` (string identity). `narrowParameters(FULL_CAPABILITIES)` deep-equals `FORMLOOM_PARAMETERS`. Every existing test in v1.2 stays green untouched.
- **Per-capability coverage.** Each row in the enforcement table gets: passing case, strict-reject case (exact error path asserted), lenient-drop case (warnings + `droppedFields` asserted).
- **Integration.** Round-trip a malicious schema through a restricted bundle; verify strict rejection and lenient drop paths end-to-end.
- **Byte-identity regression.** Added as CI gate to prevent accidental prompt drift between minor releases.

## Open questions

These are left open deliberately — call them out on this PR if you have a view:

1. **Should `createFormloomCapabilities` memoize?** The bundle is computed once per call. Consumers reconstructing per request pay the cost. Leaning "no, let userland memoize" because the function is pure.
2. **Should `maxFields` / `maxOptions` always error, even in lenient mode?** Current plan: truncate in lenient. Alternative: always hard-reject. Lean truncate for consistency with the rest of the table.
3. **Should we re-export `FormloomCapabilities` from `@formloom/llm`?** The type lives in `@formloom/schema` to avoid a cycle, but power users of the factory will import it. Planning yes (re-export for ergonomics).
4. **Should a single `createFormloomCapabilities` call be able to spit out multiple bundles** (e.g. for different providers)? Current plan has each bundle carry all five provider tools. Seems fine.
